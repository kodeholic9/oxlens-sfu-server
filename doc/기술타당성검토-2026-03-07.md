# 기술 타당성 검토서 (Technical Feasibility Review)

**프로젝트**: light-livechat SFU Server  
**작성일**: 2026-03-06  
**범위**: 뮤트 전략, 무전기(PTT) SSRC 리라이팅, 대기 단말 비용 최적화

---

## 1. 에스컬레이션(Escalation) 뮤트

### 1.1 개요

즉각적인 반응성(UX)과 하드웨어 리소스 절약(배터리)을 동시에 확보하기 위한 2단계 뮤트 전략.

### 1.2 동작 구조

| 단계      | 시점      | 동작                                        | 효과                     |
| --------- | --------- | ------------------------------------------- | ------------------------ |
| Soft Mute | 0초 ~ 5초 | `track.enabled = false`                     | 즉각 묵음, 하드웨어 유지 |
| Hard Mute | 5초 이후  | `replaceTrack(dummyTrack)` + `track.stop()` | 하드웨어 해제, RTP 유지  |

### 1.3 `track.enabled = false`의 내부 동작

- W3C 스펙: 트랙 출력을 "zero-information content"(묵음/검정 프레임)로 전환
- 트랙 객체는 `readyState: "live"` 유지, 하드웨어 캡처 중단 없음
- 오디오: 인코더에 0 샘플 입력 → DTX 활성 시 comfort noise 간헐 전송(초당 2~3패킷)
- 비디오: 검정 프레임 입력 → 첫 키프레임 이후 극소 P-프레임 생성, 프레임레이트는 불변
- RTP 패킷 흐름 유지, SSRC/RTCP SR 정상, SFU 서버에 영향 없음

### 1.4 Dummy Track의 내부 동작

- `sender.replaceTrack(dummyTrack)`: SDP 재협상 없이 RTP sender 소스만 교체
- 동일 SSRC, 동일 codec, 동일 RTP 세션 위에 이어서 전송
- 오디오 dummy: `Web Audio API (gain=0 oscillator)` → `enabled = false`와 동일한 묵음 RTP
- 비디오 dummy: `canvas.captureStream(1)` (2x2, 1fps) → 초당 1패킷 수준으로 트래픽 극소화
- **핵심**: 원본 트랙에 `track.stop()` 호출 가능 → 물리 디바이스 해제 → 배터리 절약

### 1.5 `canvas.captureStream(fps)` 동작 확인

- Canvas API의 캡처 설정이며 SDP 협상과 무관
- SDP에 `a=framerate:30`이 협상되어 있어도, 실제 전송은 소스 입력 프레임레이트를 따름
- 1fps 소스 → 인코더 입력 1fps → RTP 출력 초당 1패킷
- `replaceTrack()` 자체가 SDP-free → 2PC 아키텍처 원칙 부합

### 1.6 `replaceTrack(null)` — 채택하지 않음

- Safari 내부 상태 불안정 유발 사례 보고
- RTP 패킷이 완전히 중단 → SFU 및 수신 측 Jitter Buffer가 네트워크 끊김으로 오인
- Jitsi 오픈소스 프로젝트에서 브라우저 호환성 이슈로 고생한 이력 있음
- 검토 결과 **사용하지 않기로 결정**

### 1.7 상태 관리 — 레이스 컨디션 방지

에스컬레이션 타이밍에 unmute, 연타 등으로 상태가 꼬일 수 있는 시나리오 식별:

| 시나리오                        | 위험                                      | 대응                                                          |
| ------------------------------- | ----------------------------------------- | ------------------------------------------------------------- |
| 에스컬레이션 직전(4.9초) unmute | 타이머 콜백이 뒤늦게 발동, 마이크 꺼짐    | 타이머 콜백 진입 시 상태 가드                                 |
| 빠른 연타 (mute→unmute→mute)    | 타이머 다중 적재                          | 상태 전환 시 `clearTimeout` 선행                              |
| Hard mute에서 unmute 중 재mute  | `getUserMedia` 비동기 완료 시 상태 불일치 | 비동기 완료 시점 상태 체크, 불일치 시 획득 트랙 즉시 `stop()` |

**해법**: 3-state 상태 머신 (`UNMUTED → SOFT_MUTED → HARD_MUTED`)

핵심 원칙:

1. 타이머 ID 단일 관리, 전환 시 항상 `clearTimeout` 선행
2. 타이머 콜백/비동기 완료 시 현재 상태 가드 체크
3. 상태 불일치 시 획득 자원 즉시 해제

### 1.8 ptime 제어 한계

- ptime은 SDP offer/answer 시점 협상 값 (Opus 기본 20ms)
- 세션 중 동적 변경은 SDP 재협상 필요 → 우리 아키텍처에서 불가
- **오디오 대안**: Opus DTX (`usedtx=1`) — `sdp-builder.mjs`에서 fmtp에 명시적으로 설정 필요
- **비디오 대안**: dummy canvas track의 캡처 fps로 간접 제어 (1fps)

### 1.9 Soft Mute + Dummy Track 순차 조합 검토

- 1단계에서 `originalTrack.enabled = false` 적용
- 2단계에서 `sender.replaceTrack(dummyTrack)` 시, 원래 트랙은 sender에서 분리됨
- dummyTrack의 `enabled`는 기본값 `true`, 원래 트랙의 enabled 상태는 무영향
- 충돌 없이 자연스럽게 이어짐
- dummyTrack에 `enabled = false`를 중복 적용하면 무해하지만 의미 없음 (묵음의 묵음)

### 1.10 서버사이드 블로킹과의 관계

현재 구현: 시그널링으로 뮤트 상태를 서버에 전달 → 서버가 패킷 포워딩 차단 → 수신 클라이언트 UI 뮤트 표시

**서버 블로킹이 있어도 dummy track은 유효한 이유:**

| 관점      | 문제                                                 | dummy track의 역할                           |
| --------- | ---------------------------------------------------- | -------------------------------------------- |
| 패킷 비용 | 서버가 블로킹해도 publisher→SFU 업링크 트래픽은 발생 | 트래픽 극소화 (audio DTX 2~3pps, video 1fps) |
| 배터리    | 마이크/카메라 하드웨어 점유                          | `track.stop()`으로 디바이스 해제             |

서버가 subscriber를 보호하고, dummy track이 publisher를 보호하는 구조.

### 1.11 업계 참고

| 서비스      | 방식                                                       | 비고                                   |
| ----------- | ---------------------------------------------------------- | -------------------------------------- |
| Discord     | 네이티브 엔진 + 서버사이드 뮤트 이중 구조                  | OS 레벨 캡처 제어                      |
| Zoom        | 네이티브 엔진, OS 레벨 캡처 제어                           | 브라우저 WebRTC 미사용                 |
| Google Meet | 순수 브라우저 기반, `enabled = false` 추정                 | 장기 뮤트 시 하드웨어 해제 여부 미확인 |
| Jitsi       | `replaceTrack(null)` 사용 → 브라우저 호환성 이슈 발생 이력 | 참고 사례                              |

## 2. 무전기(PTT) SSRC 리라이팅

### 2.1 왜 필요한가

무전기 모드에서 여러 화자가 하나의 채널에서 교대로 발화. 수신 측에서 하나의 연속 스트림으로 인식해야 하므로, 서버가 가상 SSRC로 통일 + 시퀀스/타임스탬프 리라이팅 필수. 선택이 아닌 본질적 요구사항.

### 2.2 서버가 리라이팅해야 하는 항목 — 최소 범위

light-livechat의 "light" 정체성을 유지하기 위해 최소 필수 항목만 추림.

**RTP 헤더 (필수 4개 필드):**

| 필드            | 이유                                                    |
| --------------- | ------------------------------------------------------- |
| SSRC            | 가상 SSRC로 통일, 없으면 무전기 자체 불가               |
| Sequence Number | 연속성 없으면 수신 측 Jitter Buffer 깨짐                |
| Timestamp       | 불연속이면 lip-sync 깨짐, 화자 전환 시 오프셋 계산 필요 |
| Marker Bit      | 화자 전환 시 첫 패킷에 설정 → 수신 측 버퍼 플러시 힌트  |

**서버 동작 (필수):**

- 화자 전환 시 새 publisher에게 PLI 발사
- 키프레임 도착 대기 후 키프레임부터 포워딩 재개
- NACK 처리: RtpCache로 서버 직접 응답 (Phase C-1에서 이미 구현)

### 2.3 리라이팅 연산 방식 — 오프셋 기반 상대 연산

단순 `seq++` 카운터 방식은 **사용 불가**. UDP 특성상 패킷이 순서대로 도착하지 않으므로, 도착 순서대로 가상 시퀀스를 찍으면 원본의 순서 관계가 파괴됨.

**올바른 방식: 오프셋 기반 상대 연산**

화자 전환 시점에 서버가 기록하는 상태 (4개):

| 상태 값            | 설명                               |
| ------------------ | ---------------------------------- |
| `origin_base_seq`  | 새 publisher의 첫 원본 시퀀스 넘버 |
| `virtual_base_seq` | 가상 시퀀스의 현재 연속 값         |
| `origin_base_ts`   | 새 publisher의 첫 원본 타임스탬프  |
| `virtual_base_ts`  | 가상 타임스탬프의 현재 연속 값     |

**연산 공식:**

```
virtual_seq = original_seq - origin_base_seq + virtual_base_seq
virtual_ts  = original_ts  - origin_base_ts  + virtual_base_ts
```

**이 방식의 핵심:**

- 원본 스트림의 상대적 순서가 그대로 보존됨
- 패킷이 비순차 도착(101, 100, 102)해도 가상 시퀀스의 상대 위치가 유지되어 수신 측 Jitter Buffer가 정상 재정렬 가능
- 코덱 클럭레이트가 동일하면(오디오 Opus 48000Hz, 비디오 90000Hz) 오프셋 가산만으로 타임스탬프 연속성 유지
- 화자 전환마다 4개 값만 갱신하면 되므로 서버 상태가 가벼움

### 2.4 버려도 되는 것 (품질 이슈 실측 시 추가)

| 항목                    | 없을 때 영향                                       | 판단      |
| ----------------------- | -------------------------------------------------- | --------- |
| SR 재생성               | lip-sync 정밀도 저하, 재생 자체는 됨               | 후순위    |
| REMB 라우팅             | publisher가 기본 bitrate로 전송, 품질 최적화 안 됨 | 후순위    |
| FIR                     | PLI로 대체 가능                                    | 불필요    |
| abs-send-time 갱신      | TWCC 미사용 시 불필요                              | 불필요    |
| RTCP compound 규격 준수 | 브라우저가 관대하게 처리                           | 후순위    |
| SDES                    | 초기 1회면 충분                                    | 최소 처리 |
| RTCP XR                 | 품질 모니터링용                                    | 불필요    |

### 2.5 시퀀스 넘버 리라이팅의 위험성 — 화상회의에서는 절대 하지 않음

시퀀스 리라이팅 시 연쇄 복잡도:

- NACK 역매핑 필요 → 단, RtpCache가 서버에서 직접 응답하므로 이 문제는 해소됨
- RTX 재전송 패킷 재포장
- RTCP SR/RR의 extended highest sequence number 재계산
- subscriber 다수 시 각각 독립 시퀀스 카운터 관리

**결론**: 화상회의 모드에서는 원본 시퀀스 그대로 포워딩. 무전기 모드에서만 리라이팅 적용. 두 모드의 RTCP 처리 경로가 분기되어야 함.

### 2.6 서버사이드 블로킹 해제 시 수신 측 리스크

서버가 뮤트 해제 후 포워딩을 재개할 때, 수신 측에서 발생할 수 있는 문제:

**시퀀스 갭 문제:**

- RTP sequence number는 16비트 (0~65535)
- 오디오 ptime 20ms 기준: 초당 50패킷, 약 22분에 1회 wrap around
- 30분 뮤트 → 1회, 1시간 → 2~3회 오버플로우 발생
- 수신 측이 마지막 seq 1000 → 갑자기 seq 55000 수신 시, 54000개 손실로 해석 가능성

**잠재적 위험:**

- NACK 폭풍: 수신 측이 대량 패킷 손실로 판단 → 대량 NACK 발생 (RtpCache 128슬롯으로 대부분 응답 불가)
- 비디오: 블로킹 해제 후 첫 패킷이 P-프레임이면 디코딩 불가

**브라우저 보호 로직:**

- libwebrtc는 시퀀스 갭이 임계값 초과 시 "discontinuity"로 판단, Jitter Buffer 리셋
- 단, 이 임계값과 동작은 브라우저 버전/엔진별로 다름 (스펙 외 구현 의존)
- Safari, Firefox에서 동일 동작 보장 없음

**현재 대응 방향:**

- 시퀀스는 원본 그대로 포워딩 (리라이팅하지 않음)
- 블로킹 해제 시 PLI 발사 → 키프레임부터 포워딩 재개
- 시퀀스 갭 처리는 브라우저 discontinuity 감지에 위임
- **추가 검증 필요**: 장시간 뮤트(30분~1시간) 후 해제 시 Chrome/Safari/Firefox 실측 테스트

## 3. 대기 단말 비용 최적화

### 3.1 문제

무전기 채널에 대기 중인 단말은 발화하지도 수신하지도 않지만, PeerConnection이 활성 상태이면 STUN keepalive, RTCP, dummy 패킷이 지속 발생. Wi-Fi에서는 무시 가능하나, LTE/5G 데이터 망에서는 비용 문제.

### 3.2 웹 브라우저 환경의 한계

- ptime 동적 변경 불가 (SDP 재협상 필요)
- RTP 전송 주기 직접 제어 불가
- 최선의 최적화: DTX(오디오) + 1fps dummy canvas(비디오) + 서버사이드 블로킹

### 3.3 안드로이드 네이티브 환경

libwebrtc를 직접 빌드하여 사용할 경우 브라우저에서 불가능한 제어가 가능:

| 제어 항목                     | 방법                                                           |
| ----------------------------- | -------------------------------------------------------------- |
| Opus DTX 직접 제어            | `WebRtcOpus_EnableDtx()` 네이티브 API                          |
| 비디오 프레임레이트 동적 변경 | `VideoEncoder` 런타임 설정 변경                                |
| RTP 전송 일시 중지            | `RtpSender` 내부에서 패킷 전송 중단, ICE/DTLS keepalive만 유지 |
| STUN keepalive 주기 조정      | libwebrtc 포크 시 가능                                         |
| 커스텀 PacedSender            | 전송 주기 완전 제어                                            |

### 3.4 방향성

- 웹 환경: 기능 검증 및 프로토타이핑 용도. 비용 최적화는 최선 노력 수준
- 안드로이드 네이티브: 상용화 단계에서 러기드 단말 대상으로 최적화 적용
- libwebrtc 직접 빌드/유지보수 비용을 감수해야 하는 영역

## 4. 결론 및 다음 단계

### 4.1 채택 사항

| 항목                                         | 결정                                                 |
| -------------------------------------------- | ---------------------------------------------------- |
| 에스컬레이션 뮤트 (Soft → Hard)              | 채택. 5초 임계값, 3-state 상태 머신                  |
| Dummy Track (Audio: DTX, Video: 1fps canvas) | 채택. `replaceTrack(null)` 대신 사용                 |
| 서버사이드 블로킹 (컨퍼런스)                 | **불필요**. dummy track이 묵음/검정을 RTP로 전송하므로 수신 측에서 무음/무영상. 블로킹 시 해제 시점 시퀀스 갭 문제만 추가됨. |
| 무전기 SSRC 리라이팅 최소 범위               | RTP 헤더 4필드 + PLI + NACK만. 나머지는 실측 후 추가 |
| 화상회의 모드 시퀀스 리라이팅                | 하지 않음                                            |

### 4.2 실측 검증 완료 (2026-03-07)

- **에스컬레이션 뮤트 SDK 구현 완료** (v0.3.0)
  - Soft mute(5초 이내) → unmute: `track.enabled` 토글만, 인코더/SDP 영향 없음 ✅
  - Hard mute(5초 이후) → unmute: `replaceTrack(dummyTrack)` → `replaceTrack(새 트랙)`, SDP 재협상 없음, SSRC 유지 ✅
  - 인코더: hard mute 중 `encoderImplementation` 미보고(정상 — dummy track이라 인코딩 프레임 극소), unmute 후 `libvpx` 복구 ✅
  - 수신 측: dummy 구간 bitrate=0kbps, freeze 5건(hard mute 전환 시점 일시적, 예상된 동작)
- **서버사이드 블로킹: 컨퍼런스 모드에서 불필요 판정**
  - dummy track이 묵음 RTP를 흘리므로 수신 측에서 "아무 소리/영상 안 남" — mute 목적 달성
  - 블로킹을 넣으면 해제 시점 시퀀스 갭 문제(§2.6)를 추가로 처리해야 하므로 복잡도만 증가
  - 업링크 트래픽: DTX 2~3pps + 비디오 1fps = 사실상 무시 가능
  - **무전기(PTT) 모드에서는 서버사이드 블로킹이 별도 검토 필요** (SSRC 리라이팅과 연동)
