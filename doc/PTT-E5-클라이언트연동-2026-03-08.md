## 1. 작업 배경

### 1.1 시작 상태 (v0.5.1)

PTT 서버 미디어 파이프라인 구현 완료:
- Floor Timer, Relay Gate, Audio/Video SSRC Rewriting
- VP8 키프레임 대기, NACK 역매핑, PTT 메트릭 7개

미해결 2건:
1. `kf_arrived=0` — VP8 키프레임 감지 실패 의심
2. Subscribe SDP에 가상 SSRC 미선언 — 상대방 영상 미표시

### 1.2 문제의 본질

서버가 가상 SSRC로 RTP를 리라이팅하여 전송하지만, subscribe SDP에는 원본 SSRC가 선언됨.
Chrome은 `sdes:mid` extmap 제거 상태에서 SSRC 기반 demux를 사용하므로,
SDP에 없는 가상 SSRC 패킷을 별도 unnamed MediaStream으로 생성 → `<video>` 요소에 미연결.

## 2. 수정 내역

### 2.1 서버 (handler.rs) — ROOM_JOIN 응답에 가상 SSRC 추가

PTT 모드 방 입장 시 응답에 `ptt_virtual_ssrc` 필드 추가.

```json
{
  "room_id": "...",
  "mode": "ptt",
  "ptt_virtual_ssrc": { "audio": 1261093080, "video": 3431042260 },
  "server_config": { ... },
  "tracks": [ ... ]
}
```

Conference 모드에서는 이 필드 없음 (zero impact).

### 2.2 클라이언트 SDP (sdp-builder.js) — PTT subscribe SDP 분기

`buildSubscribeRemoteSdp(serverConfig, tracks, options)` 3번째 파라미터 추가.

PTT 모드에서 `buildPttSubscribeSdp()` 호출:
- Conference: publisher N명 × 2 m-line (audio + video each)
- PTT: **가상 audio 1개 + 가상 video 1개 = 2 m-line only**
- 트랙 유무와 무관하게 항상 가상 SSRC 선언 (early return보다 앞에 분기)
- `sdes:mid` extmap 제거 (SSRC 기반 demux)
- msid: `light-ptt ptt-audio`, `light-ptt ptt-video`

### 2.3 클라이언트 SDK (livechat-sdk.js, media-session.js)

- `_onJoinOk`: ROOM_JOIN 응답의 `mode`, `ptt_virtual_ssrc`를 MediaSession에 전달
- `MediaSession.setup`: `_sdpOptions` 저장
- `_setupSubscribePc`: PTT 모드일 때 트랙 없어도 subscribe PC 생성 (early return 우회)
- `buildSubscribeRemoteSdp` / `updateSubscribeRemoteSdp` 호출 시 options 전달

### 2.4 클라이언트 UI (app.js) — PTT 가상 스트림 바인딩

- `media:track` 이벤트에서 `stream.id === "light-ptt"` 감지 → `"__ptt__"` 키로 저장
- PTT video track의 `onunmute` / `onmute` 이벤트 핸들러 등록
- `updatePttView("listening")`: `remoteStreams.get("__ptt__")` 사용 (speaker별 대신)
- `updatePttView("idle")`: PTT 모드에서 `video.srcObject = null` 안 함 (스트림 유지)
- listening 상태에서 `video.play()` 강제 호출 (Chrome hidden→visible 대응)

### 2.5 서버 (handler.rs) — PLI 반복 전송

Floor Granted 시 PLI를 3회 전송 (0ms, 500ms, 1500ms).
hard_off → talking 복귀 시 getUserMedia + replaceTrack의 비동기 지연을 커버.

### 2.6 서버 (metrics/mod.rs, ingress.rs) — PTT 메트릭 5개 추가

| 메트릭 | 설명 |
|--------|------|
| `audio_rewritten` | 오디오 리라이팅 성공 수 (PT=111) |
| `video_rewritten` | 비디오 리라이팅 성공 수 (PT=96) |
| `video_skip` | 비디오 Skip 수 (화자 불일치) |
| `floor_released` | Floor Release 횟수 |
| `speaker_switches` | 화자 전환 횟수 |

어드민 SFU 패널 + 스냅샷 텍스트 표시 업데이트.

## 3. 디버깅 과정 요약

### 3.1 kf_arrived=0 문제

**초기 가설**: `is_vp8_keyframe()` 로직 오류.

**진단 과정**:
1. 진단 로그 추가 → `is_detail` 조건에 audio가 먼저 50개 소진하여 비디오 로그 미출력
2. `static AtomicU64` 비디오 전용 카운터로 변경
3. VP8 payload descriptor + bitstream 바이트까지 덤프

**결론**: `is_vp8_keyframe()` 로직 자체는 **정상**. `kf_arrived=0`은 텔레메트리 3초 윈도우 타이밍 이슈.
실제 Floor Granted 후 키프레임이 정상 감지되고 리라이팅 수행됨을 확인.

```
[DBG:VP8:KF] kf=true S=true X=true pd=0x90 xb=0x80 bs_off=28 bs=0x90 frame_type=0
[PTT:REWRITE] keyframe arrived user=U608 ssrc=0x9432870C
[PTT:REWRITE] first_pkt user=U608 orig_seq=31073 → v_base_seq=1 v_base_ts=3000
```

### 3.2 Subscribe SDP 가상 SSRC 미선언

**원인**: `buildSubscribeRemoteSdp`에서 `tracks.length === 0` early return이 PTT 분기보다 먼저 실행.
**수정**: PTT 분기를 early return보다 앞으로 이동 + `hasActiveTracks` 조건 제거.

### 3.3 Subscribe PC 미생성

**원인**: `_setupSubscribePc`에서 `activeTracks.length === 0 && !this._subPc` → early return.
**수정**: PTT 모드일 때 이 조건 우회 (`!isPtt` 추가).

### 3.4 PTT 가상 스트림 UI 바인딩 실패

**원인**: `stream.id = "light-ptt"` → `userId = "ptt"` → `remoteStreams.get("U690")` 불일치.
**수정**: PTT 스트림을 `"__ptt__"` 키로 저장, listening 상태에서 이 키 사용.

### 3.5 화자 교대 후 영상 정지 (trackMuted=true)

**원인**: idle 구간에서 서버가 RTP를 안 보내면 Chrome이 track을 muted로 전환.
다시 패킷이 와도 키프레임 없이는 디코더가 시작 못 함.

**수정**:
- idle 상태에서 `video.srcObject = null` 안 함 (PTT 모드)
- `video.play()` 강제 호출
- PLI 반복 전송 (0ms, 500ms, 1500ms)
- `onunmute` 이벤트에서 즉시 video 연결

## 4. WebRTC 기반 PTT의 본질적 한계

### 4.1 설계 철학의 불일치

WebRTC는 **연속적인 양방향 미디어 스트림**을 전제로 설계되었다.
PTT는 **간헐적인 단방향 스트림 전환**이다.

이 근본적 불일치로 인해 다음 문제들이 발생한다.

### 4.2 Chrome track muted 전이

서버가 idle 구간에서 RTP를 전송하지 않으면 (PTT 게이팅),
Chrome은 수 초 이내에 MediaStreamTrack을 `muted` 상태로 전환한다.

```
track.muted: false → true  (RTP 중단 후 ~2-3초)
track.muted: true → false  (RTP 재개 + 키프레임 디코딩 성공 후)
```

`muted → false` 전이에는 **유효한 키프레임 디코딩이 선행**되어야 하므로,
PLI 전송 → 인코더 응답 → 서버 릴레이 → 디코더 처리까지 최소 1-2 RTT + α의 지연이 발생.

### 4.3 키프레임 타이밍 문제

화자 전환 시 서버가 PLI를 보내 키프레임을 요청하지만:

1. **PLI → 키프레임 응답 지연**: 최소 1 RTT (~2-10ms LAN, ~50-200ms WAN)
2. **인코더 응답 불확실**: Chrome VP8 인코더가 즉시 키프레임을 생성하지 않을 수 있음
3. **hard_mute 복귀 시**: getUserMedia + replaceTrack 비동기 지연(300-800ms)으로
   PLI 응답 시점에 아직 dummy track이 활성 → 검은 키프레임 전송

대응: PLI 반복 전송(0ms, 500ms, 1500ms)으로 커버하지만, **완벽한 해결이 아님**.

### 4.4 idle → listening 전환 시 영상 지연

정상 시나리오에서도 화자 전환 후 첫 키프레임이 디코딩될 때까지
0.5-2초의 **영상 정지 또는 이전 프레임 표시** 구간이 존재.

이는 WebRTC의 "연속 스트림" 전제에서 벗어나는 사용 패턴의 필연적 결과.

### 4.5 Jitter Buffer 플러시 불확실

화자 전환 시 marker bit을 설정하여 Jitter Buffer 플러시를 힌트하지만,
이는 RFC 상의 **권고**이지 강제가 아니다. 브라우저 구현에 따라 동작이 다를 수 있다.

### 4.6 오디오 지연 관찰

테스트에서 오디오 지연이 관찰됨. 가능한 원인:

1. **PTT 게이팅 → 첫 패킷 지연**: Floor Granted 시그널링(WS) → RTP 통과까지 RTT
2. **Jitter Buffer 재적응**: idle 후 패킷 수신 재개 시 JB가 적응하는 시간
3. **Opus DTX**: 묵음 구간에서 comfort noise만 전송 → 수신 측 DTX 복구 지연
4. **soft_mute → live 전환**: `track.enabled = true` 후 실제 인코더 출력까지의 지연

현재 메트릭으로는 정확한 구간 특정이 어려워, audio_rewritten / video_rewritten
분리 카운팅을 추가하여 향후 분석 기반을 마련함.

## 5. PTT 오디오 vs 영상: 실용적 판단

### 5.1 핵심 가치

PTT의 본질은 **음성**이다. 영상은 부가 기능.
실제 상용 PTT 제품들의 영상 처리:

| 제품 | 영상 | 비고 |
|------|------|------|
| Zello | 별도 모드 | 음성 PTT와 영상 메시지 분리 |
| Voxer | 미지원 | 음성 전용 |
| WAVE (Motorola) | 제한적 | 별도 영상 세션 |
| Talkman | 미지원 | 산업용 음성 전용 |

### 5.2 현재 구현의 위치

| 기능 | 상태 | 비고 |
|------|------|------|
| PTT 오디오 | ✅ 작동 | 지연 관찰 → 추가 분석 필요 |
| PTT 영상 (soft_off 복귀) | ✅ 작동 | 키프레임 대기 후 정상 표시 |
| PTT 영상 (hard_off 복귀) | ⚠️ 불안정 | 검은 화면 가능 (PLI 타이밍) |
| 화자 교대 영상 전환 | ⚠️ 지연 | 0.5-2초 정지 후 동기화 |
| 3인 PTT | ✅ 작동 | 영상 포함 정상 확인 |

### 5.3 향후 방향 제안

**단기 (v0.5.x)**:
- PTT 오디오 지연 원인 분석 및 최적화
- hard_mute 에스컬레이션 타이머 조정 (60초 → 환경 설정화)
- 영상은 "best effort" — 키프레임 도착까지 이전 프레임 표시

**중기 (v0.6.x)**:
- 서버 키프레임 캐싱 — 화자 전환 시 마지막 키프레임 즉시 주입 (LiveKit 패턴)
- PLI 불필요 → 전환 지연 0에 수렴

**장기**:
- PTT 영상을 Conference 모드와 분리된 독립 기능으로 재설계
- 또는 PTT = 음성 전용으로 단순화, 영상은 Conference 모드에서만 제공
