## 읽는 순서 (권장)

소스를 처음 볼 때 **파일 순서가 곧 이해 순서**다.
아래 번호 순서대로 읽으면 의존성이 꼬이지 않는다.

```
1단계: 뼈대          →  main.rs → lib.rs → config.rs → error.rs → state.rs
2단계: 시그널링       →  signaling/opcode.rs → message.rs → handler.rs
3단계: 방/참가자      →  room/room.rs → participant.rs
4단계: 네트워크       →  transport/demux.rs → stun.rs → ice.rs → dtls.rs → srtp.rs → demux_conn.rs
5단계: 미디어 파이프라인  →  transport/udp/mod.rs → ingress.rs → egress.rs → rtcp.rs → twcc.rs
6단계: PTT           →  room/floor.rs → ptt_rewriter.rs
7단계: 관측           →  metrics/mod.rs → env.rs → tokio_snapshot.rs
```

## 1단계: 뼈대 (5파일, 28KB)

### main.rs (214B) — 여기서 시작

```rust
#[tokio::main]
async fn main() {
    oxlens_sfu_server::run_server().await  // lib.rs의 run_server() 호출
}
```

진짜 1줄짜리. `run_server()`가 모든 걸 한다.

### lib.rs (18KB) — 서버 조립 공장

JavaScript로 치면 `app.js` (Express 서버 setup). 하는 일:

1. `.env` 파일 읽기 (Vue CLI의 `.env.local` 같은 것)
2. 로그 시스템 초기화
3. DTLS 인증서 생성 (HTTPS 인증서와 비슷, 매 실행마다 새로 만듬)
4. UDP 소켓 바인딩 (SO_REUSEPORT로 멀티 워커)
5. `GlobalMetrics` 생성 (전역 카운터 객체)
6. `AppState` 조립 (모든 공유 상태를 하나로)
7. 기본 방 3개 생성 ("무전 대화방", "회의실-2", "대회의실")
8. UDP 워커 N개 spawn (패킷 수신 루프)
9. 좀비 리퍼 태스크 spawn (30초마다 죽은 세션 정리)
10. 발화권 타이머 태스크 spawn (PTT용, 2초마다)
11. Axum 웹서버 시작 (`/ws`, `/admin/ws` 두 개 엔드포인트)
12. Ctrl+C → graceful shutdown

**Vue 개발자 관점**: Express의 `app.listen()` 전에 미들웨어 세팅하는 것과 같다.

### config.rs (6KB) — 모든 상수의 집합

매직 넘버 없이 여기서 관리. JavaScript의 `const config = { ... }` 같은 파일.

주요 상수:
- `WS_PORT: 1974`, `UDP_PORT: 19740`
- `HEARTBEAT_INTERVAL_MS: 30_000`, `HEARTBEAT_TIMEOUT_MS: 90_000`
- `ZOMBIE_TIMEOUT_MS: 120_000`
- `RTP_CACHE_SIZE: 512` (NACK→RTX 재전송 캐시)
- `EGRESS_QUEUE_SIZE: 256` (구독자당 큐)
- `FLOOR_MAX_BURST_MS: 30_000` (PTT 최대 발화 시간)
- `RoomMode` enum: `Conference` / `Ptt`
- `BweMode` enum: `Twcc` / `Remb`

### error.rs (2KB) — 에러 코드 체계

```
1xxx: 연결/인증 (NotAuthenticated, InvalidToken, ...)
2xxx: 방 (RoomNotFound, RoomFull, AlreadyInRoom, ...)
3xxx: 시그널링 (InvalidOpcode, InvalidPayload, ...)
4xxx: 미디어/전송 (SdpParseError, DtlsHandshakeFailed, SrtpError)
9xxx: 서버 내부
```

`LightError` enum + `LightResult<T>` 타입 별칭.
JavaScript의 `class AppError extends Error` 패턴.

### state.rs (2KB) — 전역 공유 상태

```rust
pub struct AppState {
    pub rooms:       Arc<RoomHub>,        // 모든 방
    pub cert:        Arc<ServerCert>,      // DTLS 인증서
    pub udp_socket:  Arc<UdpSocket>,       // PLI 전송용
    pub metrics:     Arc<GlobalMetrics>,   // 전역 카운터
    pub admin_tx:    broadcast::Sender,    // 어드민 WS 채널
    pub public_ip:   String,              // ICE candidate IP
    pub ws_port / udp_port / bwe_mode / remb_bitrate ...
}
```

**Vue 개발자 관점**: Vuex store 루트와 같다. 모든 핸들러가 이걸 주입받는다.

## 2단계: 시그널링 (50KB)

### opcode.rs (2KB) — 프로토콜 상수표

```
Client → Server:  HEARTBEAT(1), IDENTIFY(3), ROOM_JOIN(11), PUBLISH_TRACKS(15), ...
Server → Client:  HELLO(0), ROOM_EVENT(100), TRACKS_UPDATE(101), ...
Floor Control:    FLOOR_REQUEST(40), FLOOR_TAKEN(141), FLOOR_REVOKE(143), ...
```

디스코드 Gateway 프로토콜과 같은 패턴 (`{"op": N, "d": {...}}`).

### message.rs (4KB) — 패킷 구조체

```rust
struct Packet { op, pid, ok, d }       // 모든 WS 메시지의 공통 포맷
struct IdentifyRequest { token }
struct RoomJoinRequest { room_id }
struct PublishTracksRequest { tracks: Vec<{kind, ssrc}> }
struct FloorRequestMsg { room_id }
...
```

**Vue 개발자 관점**: API의 Request/Response DTO와 같다.

### handler.rs (44KB) — ★ 비즈니스 로직의 심장

**가장 크고 가장 중요한 파일.** WS 연결의 전체 생명주기를 관리한다.

#### 핵심 구조

```
ws_handler()              — Axum HTTP → WebSocket 업그레이드
  └─ handle_connection()  — WS 연결 하나의 전체 수명
       ├─ HELLO 전송 (heartbeat_interval 안내)
       ├─ select 루프 (3갈래: WS 수신 / 송신 큐 / 하트비트 타이머)
       │   └─ dispatch() — opcode별 분기
       │        ├─ IDENTIFY   → handle_identify()
       │        ├─ ROOM_LIST  → handle_room_list()
       │        ├─ ROOM_CREATE → handle_room_create()
       │        ├─ ROOM_JOIN  → handle_room_join()      ★ 가장 복잡
       │        ├─ PUBLISH_TRACKS → handle_publish_tracks()
       │        ├─ MUTE_UPDATE → handle_mute_update()
       │        ├─ ROOM_LEAVE → handle_room_leave()
       │        ├─ MESSAGE    → handle_message()
       │        ├─ FLOOR_REQUEST/RELEASE/PING → PTT 핸들러
       │        └─ TELEMETRY  → handle_telemetry()
       └─ cleanup()       — 연결 종료 시 정리
```

#### handle_room_join() — 가장 핵심적인 함수 (120줄)

1. ICE 자격증명 2쌍 생성 (publish + subscribe)
2. `Participant` 생성 (2PC 세션 포함)
3. `RoomHub`에 등록 (3중 인덱스 × 2세션 = 6개 키)
4. 기존 참가자들에게 `participant_joined` 브로드캐스트
5. `server_config` JSON 응답 (ICE, DTLS, 코덱 정책, extmap)
6. PTT 모드면 `ptt_virtual_ssrc` 추가

응답 예시:
```json
{
  "room_id": "...",
  "participants": ["userA", "userB"],
  "server_config": {
    "ice": { "publish_ufrag", "subscribe_ufrag", "ip", "port" },
    "dtls": { "fingerprint", "setup": "passive" },
    "codecs": [...],
    "extmap": [...]
  }
}
```

#### cleanup() — 연결 종료 정리 (80줄)

WS 끊기면 자동 실행:
1. Floor 발화권 자동 해제 (PTT)
2. 방에서 참가자 제거
3. 남은 참가자에게 `tracks_update(remove)` + `participant_left` 브로드캐스트

## 3단계: 방/참가자 (44KB)

### room.rs (13KB) — Room + RoomHub

**3중 인덱스 구조** (JavaScript의 3개 Map을 동시에 관리):

```
Room
  ├── participants: DashMap<user_id, Participant>     ← 시그널링용
  ├── by_ufrag:     DashMap<ufrag, (Participant, PcType)>  ← STUN용
  └── by_addr:      DashMap<addr, (Participant, PcType)>   ← SRTP 핫패스용

RoomHub (전역)
  ├── rooms:       DashMap<room_id, Room>
  ├── ufrag_index: DashMap<ufrag, room_id>       ← STUN → 방 찾기
  └── addr_index:  DashMap<addr, room_id>        ← SRTP → 방 찾기
```

**왜 3중 인덱스?**
- 시그널링: `user_id`로 참가자 찾기
- STUN 패킷: `ufrag`로 참가자 찾기 (연결 초기)
- SRTP 패킷: `IP:Port`로 참가자 찾기 (미디어 전송 중, 초당 수천번)

**DashMap**: `HashMap`의 동시성 버전. JavaScript의 `Map`인데 여러 스레드가 동시에 읽기/쓰기 가능.

핵심 메서드:
- `latch()` — STUN 성공 → IP:Port 등록 (NAT rebinding 대응)
- `find_by_addr()` — O(1) 참가자 조회 (hot path)
- `reap_zombies()` — 타임아웃 참가자 제거

### participant.rs (12KB) — 2PC 세션 모델

**2PC = PeerConnection 2개** (publish + subscribe 분리):

```
Participant
  ├── user_id, room_id, joined_at, last_seen
  ├── ws_tx: UnboundedSender<String>        ← WS로 메시지 전송
  │
  ├── publish: MediaSession                  ← 클라이언트 → 서버
  │     ├── ufrag, ice_pwd
  │     ├── address: Mutex<Option<SocketAddr>>
  │     ├── inbound_srtp                     ← 수신 복호화
  │     └── outbound_srtp                    ← 사용 안 함 (PLI 등만)
  │
  ├── subscribe: MediaSession                ← 서버 → 클라이언트
  │     ├── ufrag, ice_pwd
  │     ├── address
  │     ├── inbound_srtp                     ← subscribe RTCP 복호화
  │     └── outbound_srtp                    ← 미디어 암호화 후 전송
  │
  ├── tracks: Mutex<Vec<Track>>              ← 이 참가자가 보내는 트랙들
  ├── rtp_cache: Mutex<RtpCache>             ← NACK→RTX용 링버퍼 (512슬롯)
  ├── twcc_recorder                          ← TWCC 도착시간 기록
  ├── egress_tx / egress_rx                  ← subscriber egress 채널
  └── rtx_seq: AtomicU16                     ← RTX 패킷 시퀀스
```

**Vue 개발자 관점**: 한 사용자의 모든 WebRTC 상태를 담는 거대한 Vuex 모듈.

## 4단계: 네트워크 전송 (31KB)

### demux.rs (2KB) — 패킷 분류기

```
UDP 패킷 도착 → 첫 바이트 확인:
  0x00~0x03 → STUN (ICE 연결 수립)
  0x14~0x3F → DTLS (암호화 핸드셰이크)
  0x80~0xBF → SRTP (미디어 데이터)
```

### stun.rs (13KB) — STUN 프로토콜 (RFC 8489)

Binding Request 파싱 → MESSAGE-INTEGRITY 검증 → Binding Response 생성.
UDP 홀펀칭의 기반. Chrome이 "이 서버에 연결할 수 있니?" 확인하는 프로토콜.

### ice.rs (4KB) — ICE-Lite

서버는 passive. 자격증명(ufrag+pwd) 생성만 하고, 클라이언트가 보낸 STUN에 응답만 한다.

### dtls.rs (5KB) — DTLS 핸드셰이크

HTTPS의 TLS와 같은 것이지만 UDP용. `webrtc-rs` 라이브러리 래핑.
핸드셰이크 완료 후 SRTP 키를 추출한다 (RFC 5764).

### srtp.rs (5KB) — SRTP 암복호화

`webrtc-srtp` 라이브러리 래핑. encrypt/decrypt 4가지:
- `decrypt_rtp()` / `encrypt_rtp()` — 미디어 패킷
- `decrypt_rtcp()` / `encrypt_rtcp()` — 제어 패킷

### demux_conn.rs (3KB) — 가상 연결 어댑터

UDP 소켓 하나를 여러 DTLS 세션이 공유하기 위한 브릿지.
mpsc 채널로 패킷을 DTLS 라이브러리에 전달.

## 5단계: 미디어 파이프라인 (93KB) — ★ 성능 핵심

### udp/mod.rs (17KB) — UDP 메인 루프

```
loop {
    tokio::select! {
        // 1. UDP 패킷 수신
        Ok((len, remote)) = socket.recv_from(&mut buf) => {
            match demux::classify(&buf) {
                STUN → STUN 처리 + DTLS 트리거
                DTLS → DTLS 세션에 패킷 전달
                SRTP → ingress.handle_srtp()  ★ hot path
            }
        }
        // 2. 100ms 타이머: TWCC feedback 전송
        _ = bwe_timer.tick() => { send_twcc/remb }
        // 3. 3초 타이머: 메트릭 flush
        _ = metrics_timer.tick() => { flush_metrics }
    }
}
```

### udp/ingress.rs (25KB) — ★ 수신 hot path

```
handle_srtp(buf, remote):
  1. find_by_addr(remote) → (Participant, PcType, Room) 찾기  O(1)
  2. PcType::Publish만 처리 (subscribe는 무시)
  3. inbound_srtp.decrypt()  → plaintext
  4. RTP 헤더 파싱 (SSRC, PT, seq, ts)
  5. VP8(video)면 RtpCache에 저장 (NACK→RTX용)
  6. TWCC 도착시간 기록
  7. [PTT] floor 게이팅 (발화자만 통과)
  8. [PTT] SSRC 리라이팅 (audio/video)
  9. fan-out: room 내 다른 참가자들에게 → egress_tx.try_send()
```

패킷 하나당 이 함수가 1번 실행된다.
30명 회의에서 초당 수천 번 호출되므로 **성능이 가장 중요한 코드**.

### udp/egress.rs (8KB) — 구독자별 송신 태스크

```
run_egress_task(participant):
  loop {
    packet = egress_rx.recv()  // ingress에서 보낸 plaintext
    encrypted = outbound_srtp.encrypt(packet)
    socket.send_to(encrypted, subscribe_addr)
  }
```

구독자마다 독립 태스크. outbound_srtp Mutex 경합 제거 (Phase W-3 핵심).

### udp/rtcp.rs (13KB) — RTCP 파서/빌더

NACK 파싱, RTX 조립, PLI 빌더, REMB 빌더 등 유틸리티 모음.
WebRTC 제어 프로토콜의 "도구 상자".

### udp/twcc.rs (30KB) — TWCC 혼잡 제어

Chrome이 보내는 RTP에 포함된 전송 시퀀스 번호를 기록하고,
100ms마다 "패킷 도착 시간 보고서"를 Chrome에 보낸다.
Chrome은 이걸 보고 비트레이트를 자율 조절한다.

## 6단계: PTT (19KB)

### floor.rs (8KB) — 발화권 상태 머신

```
FloorState:  Idle ←→ Taken { speaker, burst_start, last_ping }

FloorAction:
  Granted    → 발화권 획득 (PTT 누름)
  Denied     → 이미 누가 쓰는 중
  Released   → 자진 해제 (PTT 뗌)
  Revoked    → 서버 강제 회수 (30초 초과 또는 PING 미수신)
```

3GPP MCPTT 표준 간소화 버전.

### ptt_rewriter.rs (11KB) — SSRC 리라이팅

PTT 모드에서 여러 화자의 RTP를 **하나의 가상 스트림**으로 통합:

```
화자A의 SSRC=1234, seq=100,101,102...
화자B의 SSRC=5678, seq=200,201,202...

→ 가상 SSRC=9999로 통합:
  seq=1,2,3 (화자A) → seq=4,5,6 (화자B) → 연속!
```

수신측 Chrome이 단일 스트림으로 인식하도록 서버가 **RTP 헤더를 in-place로 덮어쓴다**.

## 7단계: 관측 (21KB)

### metrics/mod.rs (16KB) — GlobalMetrics

모든 카운터를 `AtomicU64`로 관리 (Mutex 없음, lock-free).
3초마다 `flush()` → JSON → 어드민 WS로 전송.

### metrics/env.rs (1KB) — 환경 메타

빌드 모드, 로그 레벨, 워커 수, BWE 모드, 버전. 시작 시 1회 결정.

### metrics/tokio_snapshot.rs (4KB) — Tokio 런타임 지표

3초마다 Tokio 내부 지표를 스냅샷: busy_ratio, alive_tasks, global_queue 등.

## 핵심 데이터 흐름 3가지

### 1. 방 입장 흐름 (시그널링)

```
[Browser] → WS IDENTIFY(token) → [handler] → user_id 확정
[Browser] → WS ROOM_JOIN(room_id) → [handler]
   → ICE 자격증명 2쌍 생성
   → Participant 생성 (publish + subscribe 세션)
   → RoomHub에 3중 인덱스 등록
   → server_config JSON 응답
[Browser] → SDP 조립 (클라이언트 측) → ICE 연결 시작
```

### 2. 미디어 전송 흐름 (hot path)

```
[Browser] → UDP(SRTP) → [udp/mod.rs recv_from]
   → demux → SRTP
   → [ingress.rs handle_srtp]
      → find_by_addr O(1) → Participant
      → decrypt → plaintext
      → rtp_cache.store (video)
      → twcc_recorder.record
      → [PTT] floor gate + rewrite
      → fan-out: egress_tx.try_send(plaintext) × (N-1)
   → [egress.rs run_egress_task] (per subscriber)
      → encrypt → send_to(subscribe_addr)
→ [Other Browser] 수신
```

### 3. PTT 발화권 흐름

```
[UserA] → FLOOR_REQUEST → [handler]
   → FloorController.request() → Granted
   → PttRewriter.switch_speaker("UserA")
   → FLOOR_TAKEN 브로드캐스트

[UserA 미디어] → ingress → floor gate 통과 → rewrite(가상 SSRC) → fan-out

[UserA] → FLOOR_RELEASE → [handler]
   → FloorController.release() → Released
   → PttRewriter.clear_speaker()
   → FLOOR_IDLE 브로드캐스트
```

## Rust 패턴 → JavaScript 대응표

| Rust | JavaScript/Vue 대응 | 용도 |
|------|---------------------|------|
| `Arc<T>` | 참조 카운팅 (JS는 GC가 대신) | 여러 곳에서 공유 |
| `Mutex<T>` | `let lock = await mutex.acquire()` | 동시 접근 방지 |
| `DashMap<K,V>` | `new Map()` + 읽기/쓰기 잠금 | 동시성 안전 Map |
| `AtomicU64` | `volatile` 변수 (JS에 없음) | lock-free 카운터 |
| `tokio::spawn` | `setTimeout(fn, 0)` or Worker | 비동기 태스크 생성 |
| `tokio::select!` | `Promise.race()` | 여러 비동기 중 먼저 완료된 것 |
| `mpsc::channel` | EventEmitter or MessagePort | 태스크 간 메시지 전달 |
| `broadcast::channel` | `EventBus.$emit()` | 1:N 브로드캐스트 |
| `impl Trait` | interface (TypeScript) | 다형성 |
| `enum + match` | switch-case (더 강력한 버전) | 패턴 매칭 |
| `Result<T, E>` | `try-catch` (컴파일 타임 강제) | 에러 처리 |
| `Option<T>` | `value ?? null` | null 안전 |
| `#[cfg(test)]` | `describe('test', ...)` | 테스트 코드 분리 |
| `pub(crate)` | 모듈 내부 export (외부 비공개) | 가시성 제어 |
