## 1. 금일 완료 작업 (v0.5.3)

### 1.1 PTT Lock(긴급발언) 버튼 동기화 버그 수정

- **증상:** 긴급발언 버튼 토글 상태가 실제 floor FSM 상태와 어긋남
- **원인:** `floor:denied`, `floor:idle` 등 전이 시 `pttLocked` 리셋 누락
- **수정:**
  - `floor:state` 핸들러에서 `pttLocked` 일괄 동기화 (TALKING/REQUESTING 아닌 상태 → 자동 해제)
  - `resetPttLockBtn()` 헬퍼 추출 (3곳 중복 제거)
  - `floor:revoke` 핸들러의 중복 리셋 코드 제거
- **설계 원칙 확인:** 일반발언(push-hold)과 긴급발언(toggle)은 `floorRequest()`/`floorRelease()` 공유. 트리거 방식만 다름.

### 1.2 `serde_json::json!` 매크로 재귀 한계 해소

- **증상:** `cargo build` 시 `recursion limit reached` 에러 (metrics/mod.rs)
- **원인:** 단일 `json!` 블록에 필드 ~50개 → 매크로 재귀 깊이 128 초과
- **수정:** `counters_json`, `ptt_json`, `root` 3분할 조립 + flat merge
- **JSON 출력 구조 변경 없음** — 어드민 대시보드 호환 유지

### 1.3 `maxBitrate` 서버 전달

- **설계:** 서버 `.env REMB_BITRATE_BPS` 값을 ROOM_JOIN 응답 `server_config.max_bitrate_bps`로 전달
- **클라이언트:** `serverConfig.max_bitrate_bps` 우선 적용, 없으면 클라이언트 기본값 fallback
- **효과:** 서버 재시작만으로 대역폭 정책 변경 가능. 클라이언트 재배포 불필요.

### 1.4 텔레메트리 분석 (3인 Conference)

- **결론:** 영상 멈춤의 원인은 **PC CPU 부족 → Chrome 인코더 fps 저하 → subscriber freeze**
- **SFU 서버 영역은 정상** (decrypt 0.01ms, egress_drop=0, tokio busy 4.3%)

## 2. "진정한 무전기" — WebRTC 한계 극복 논의

### 2.1 현재 WebRTC PTT의 근본 문제

화자 전환 시 첫 음성/영상 도달까지 **800ms~1.5초**. 상용 PTT 앱(Zello, Voxer)은 150~250ms.

| 구간                      | 현재 지연     | 목표                |
| ------------------------- | ------------- | ------------------- |
| Floor 시그널링            | 50~200ms      | 50ms                |
| 미디어 트랙 활성화        | 0~500ms       | 0ms (soft_off 유지) |
| 인코더 첫 프레임          | 20~50ms       | 20ms                |
| **Jitter Buffer (NetEQ)** | **300~600ms** | **50~100ms**        |
| 네트워크 전파             | 5~180ms       | 제어 불가           |

**핵심 병목:** NetEQ jitter buffer의 보수적 동작.

### 2.2 기술 선택지 검토

**Rust webrtc-rs:**

- ICE/DTLS/SRTP은 우수 (서버에서 검증 완료)
- **NetEQ 없음, AEC 없음, HW 코덱 가속 없음**
- 음성/영상 품질을 담보할 수 없음 → **탈락**

**libwebrtc (C++) 활용:**

- 구글 20년 노하우 (NetEQ, AEC, HW 가속, GCC BWE) 전부 포함
- 설정 튜닝 + 최소 패치로 PTT 최적화 가능

**결론:** libwebrtc를 미디어 엔진으로 사용하고, Rust로 PTT 비즈니스 로직을 구현.

### 2.3 네이티브 PTT 클라이언트 요구사항

**플랫폼:** iOS / Android / PC (웹 미지원)

| #   | 요구사항                                  | 해결 방향                                    |
| --- | ----------------------------------------- | -------------------------------------------- |
| 1   | 빠른 미디어 장치 구동 + 전송              | 디바이스 프리워밍 + soft mute 패턴           |
| 2   | JB 바보 방지 (패킷 공백 후 과도한 버퍼링) | NetEQ `SetMaximumDelay` + 화자 전환 시 flush |
| 3   | OS 오디오 세션 선점                       | 방 입장 시 잡고 안 놓기 (iOS/Android)        |
| 4   | 카메라 프리워밍                           | 기존 soft_off 패턴 이식                      |
| 5   | 네트워크 전환 대응 (LTE↔WiFi)             | ICE continual gathering                      |
| 6   | ICE binding 전송 주기 설정                | libwebrtc IceConfig 파라미터 제어            |

### 3.2 네이밍 — 0xLENS 브랜드 통일

"LENS로 보고, LENS로 연결한다" — 관측성(텔레메트리)과 통신(SFU) 양쪽을 커버.

| 현재 이름                   | 새 이름               | 역할                                           |
| --------------------------- | --------------------- | ---------------------------------------------- |
| `light-livechat`            | **`oxlens-sfu`**      | Rust SFU 서버                                  |
| `insight-lens` (클라이언트) | **`oxlens-sdk-web`**  | 웹 클라이언트 SDK                              |
| `insight-lens` (어드민)     | **`oxlens-admin`**    | 어드민 대시보드                                |
| (신규)                      | **`oxlens-sdk-core`** | 네이티브 클라이언트 SDK (libwebrtc + Rust PTT) |
| 도메인                      | **`oxlens.com`**      | 이미 확보                                      |

### 3.3 핵심 결정: LiveKit Rust 바인딩 활용

**LiveKit `rust-sdks` (Apache 2.0)** 가 libwebrtc → Rust FFI 바인딩을 이미 제공:

- `webrtc-sys`: libwebrtc C++ → Rust unsafe 바인딩 (cxx 기반)
- `livekit-webrtc`: Safe Rust 래퍼
- `webrtc-sys-build`: **프리빌드 libwebrtc 바이너리 자동 다운로드 + 링크**
- 플랫폼: Windows / Mac / Linux / iOS / Android 지원
- HW 코덱 가속 포함 (VideoToolbox, NVidia)
- 최신 버전 v0.3.25 (2026-02), 활발 유지보수 중

**라이선스:** Apache 2.0 — 상업적 사용, 수정, 배포 전부 자유. 소스 공개 의무 없음.
의무: LICENSE/NOTICE 파일 포함 + 변경 사항 명시 + "LiveKit" 상표 미사용.

**최대 장점: libwebrtc를 직접 빌드하지 않아도 됨.**
`cargo build` 한 번이면 프리빌드 바이너리가 자동으로 내려와서 링크됨.

## 4. libwebrtc PTT 최적화 파라미터 조사 결과

### 4.1 Level 0: 브라우저 표준 API (참고용, 네이티브에서는 Level 1 사용)

- `RTCRtpReceiver.jitterBufferTarget = 0` — Chrome 114+ 지원, hint 방식
- `playout-delay` RTP 확장 헤더 — SFU에서 삽입, Chrome이 해석

### 4.2 Level 1: libwebrtc 네이티브 API (설정만, 패치 없음) ← 1차 목표

| API                                      | 용도             | PTT 활용                |
| ---------------------------------------- | ---------------- | ----------------------- |
| `NetEq::Config.enable_fast_accelerate`   | 공격적 버퍼 축소 | true 설정               |
| `NetEq::Config.max_packets_in_buffer`    | 최대 버퍼 크기   | 200→30~50               |
| `NetEq::Config.max_delay_ms`             | 적응형 목표 상한 | 2000→300~500            |
| `NetEq::SetMaximumDelay()`               | **런타임** 제어  | 화자 전환 시 100ms 제한 |
| `NetEq::SetMinimumDelay()`               | **런타임** 제어  | 0ms 유지                |
| `AudioProcessing.echo_canceller.enabled` | AEC 비활성화     | PTT는 half-duplex       |
| `VideoReceiveStream::Config`             | 비디오 JB 설정   | max_packets=5           |

### 4.3 Level 2: 소스 패치 (Phase 4에서 필요 시)

- NetEQ 버퍼 flush API (화자 전환 시 잔여 패킷 제거)
- 비디오 JB fast drain

### 4.4 보너스 발견

- `neteq` Rust 크레이트 (lib.rs, 2025-11) — libwebrtc NetEQ 알고리즘 순수 Rust 재구현. WASM 지원. 장기적 참고용.

| 3    | libwebrtc 튜닝 API 바인딩 추가 (`webrtc-sys` fork) | 3~5일 | NetEQ/ICE 설정. **가장 기술적 리스크 큼**        |

- **단계 3 (튜닝 API 바인딩):** LiveKit이 NetEQ 내부 API까지 노출했을 가능성 낮음. `webrtc-sys` fork에서 cxx 브릿지 확장 필요.
- **프리빌드 바이너리 제약:** Level 2 소스 패치 진행 시 libwebrtc 직접 빌드 필요 (그때 추가 1~2주).
